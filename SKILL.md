# HTML Animation Skill

Create production-quality, self-contained HTML/CSS/JS animations. Every animation should look like it was crafted by a motion designer — not generated by AI.

## Before writing code

Think through these decisions first:

1. **Story arc** — What state does the viewer start in? What changes? What do they end on? Define the beginning, middle, and end.
2. **Mood** — Dark and dramatic? Clean and corporate? Playful? This dictates palette, timing, and motion style. Pick one.
3. **What earns its place** — Every element on screen must justify being there. If it doesn't help tell the story, cut it. Text especially — if a narrator or caption could say it, don't put it on screen.

## Color system

Never hardcode colors. Define a palette as CSS custom properties on `:root` and use only those.

A strong palette has:
- A background tone
- 2–3 surface layers for depth (base → elevated → most elevated)
- One dominant accent color
- Semantic colors: success (green), warning (amber), error (red)
- Muted text color with sufficient contrast

Always support light and dark themes via `[data-theme="light"]` on `<html>`:

```css
:root {
  --bg: #0f0f0f;
  --surface: #1a1a1a;
  --surface2: #1f1f1f;
  --surface3: #272727;
  --border: #272727;
  --border2: #3f3f3f;
  --text: #e4e4e7;
  --muted: #8c8c8c;
  --accent: #3b82f6;
  --accent-dim: #3b82f622;
  --green: #22c55e;
  --green-dim: #22c55e22;
  --amber: #d97706;
  --amber-dim: #d9770622;
  --red: #f87171;
  --red-dim: #f8717122;
  --shadow: 0 20px 60px rgba(0,0,0,0.5);
}

[data-theme="light"] {
  --bg: #f2f2f2;
  --surface: #ffffff;
  --surface2: #f7f7f7;
  --surface3: #ebebeb;
  --border: #d4d4d4;
  --border2: #c0c0c0;
  --text: #1a1a1a;
  --muted: #666666;
  --accent: #2563eb;
  --accent-dim: #2563eb12;
  --green: #16a34a;
  --green-dim: #16a34a12;

  --amber-dim: #d9770612;
  --red: #dc2626;
  --red-dim: #dc262612;
  --shadow: 0 20px 60px rgba(0,0,0,0.08);

```

Adapt the accent color to the project context if one is provided. The palette above is a sensible default.

### Color semantics

- **accent:** identity, interactive elements, CTAs, the "primary" thing in any visualization
- **green:** success, completion, positive outcomes
- **amber:** warnings, legacy/old state, caution
- **red:** failures, errors, critical states
- **surface layers:** depth — surface is base, surface3 is most elevated

## Typography

- UI text: `'Segoe UI', system-ui, -apple-system, sans-serif`
- Labels, data, counters, technical text: `monospace`
- No external font loading — animations must be fully self-contained
- Use `text-transform: uppercase` with `letter-spacing: 1-2px` sparingly on labels for a designed feel
- Establish a clear size hierarchy: primary content → labels → metadata

## Layout

- Center the animation: `display: flex; align-items: center; justify-content: center; min-height: 100vh`
- Use fixed pixel dimensions — don't make animations responsive
- Generous padding and gaps — cramped layouts look amateur
- Consistent `border-radius` scale: 3-4px for small elements, 6-8px for cards, 10-12px for containers
- Subtle `box-shadow` adds depth: use `var(--shadow)` from the palette
- Layer surfaces with 1px borders using `var(--border)` to create hierarchy
- Never place content directly on the background without a surface container

## Motion standards

### Entrances

Elements animate in, never just appear:

```css
.element {
  opacity: 0;
  transform: translateY(8px);
  animation: slideUp 0.6s ease 0.3s forwards;
}
@keyframes slideUp {
  to { opacity: 1; transform: translateY(0); }
}
```

Vary the delay to stagger sequential elements. Vary the translate distance (8px–16px) by element size.

### State transitions

Always use CSS transitions for visual changes:

```css
.element {
  transition: background 0.4s ease, transform 0.3s ease, opacity 0.3s ease;
}
```

Never switch states instantly. Even 200ms of transition makes a difference.

### Easing

- `ease` — default for most transitions
- `cubic-bezier(0.4, 0, 0.2, 1)` — smooth deceleration for elements settling into position
- `cubic-bezier(0.34, 1.56, 0.64, 1)` — springy/bouncy reveals (use sparingly)
- Never use `linear` for UI motion

### Timing

- **Initial delay:** 300–800ms before anything moves (let the scene load)
- **Stagger:** 60–150ms between sequential items appearing
- **Transition duration:** 300–600ms for most elements
- **End hold:** 1–2 seconds of stillness after the last animation
- **Total duration:** be deliberate — every second should have purpose

### Sequencing with JavaScript

Use `setTimeout` for orchestrating phases. Structure timing as named phases:

```javascript
// Phase 1: Elements appear (0–1s)
setTimeout(() => { /* first group */ }, 300);
setTimeout(() => { /* second group */ }, 600);

// Phase 2: State changes (1–3s)
setTimeout(() => { /* transition */ }, 1500);

// Phase 3: Resolution (3–5s)
setTimeout(() => { /* final state */ }, 3500);
```

Always comment the phase structure.

## File structure

Every animation is a single self-contained HTML file:

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <meta name="h2v-duration" content="Xs">
  <meta name="h2v-themes" content="dark,light">
  <title>Descriptive title</title>
  <style>
    /* :root palette (dark + light) */
    /* Reset */
    /* Body: centered flex, full viewport, var(--bg) */
    /* Controls bar */
    /* Animation-specific styles */
  </style>
</head>
<body>
  <div class="controls" data-h2v-hide>
    <button class="ctrl-btn" onclick="location.reload()">↺ Reset</button>
    <button class="ctrl-btn" id="themeBtn" onclick="toggleTheme()">☀ Light</button>
  </div>

  <!-- Animation content -->

  <script>
    function toggleTheme() {
      const l = document.documentElement.getAttribute('data-theme') === 'light';
      document.documentElement.setAttribute('data-theme', l ? 'dark' : 'light');
      document.getElementById('themeBtn').textContent = l ? '☀ Light' : '🌙 Dark';
    }
    window.addEventListener('message', function(e) {
      if (e.data && e.data.theme) document.documentElement.setAttribute('data-theme', e.data.theme);
    });

    // Animation logic
  </script>
</body>
</html>
```

### Controls bar CSS

```css
.controls {
  position: fixed; top: 16px; right: 16px;
  display: flex; gap: 6px; z-index: 100;
}
.ctrl-btn {
  padding: 8px 16px; background: var(--surface);
  border: 1px solid var(--border); border-radius: 8px;
  color: var(--muted); font-size: 12px; cursor: pointer;
  font-family: monospace;
}
.ctrl-btn:hover { color: var(--text); border-color: var(--border2); }
```

Every animation includes the controls bar and theme toggle. The `postMessage` listener enables theme control from a parent page embedding the animation in an iframe.

## Video export

Animations from this skill can be rendered to MP4 with the `h2v` CLI. The project lives at https://github.com/drewharvey/html-to-video — do not re-fetch that README each time; this section captures the interface you need. (`h2v` is under active development; if a command here errors in a way that suggests the CLI surface has changed, *then* check the repo README and update this section.)

**Conversion is slow — never run it as part of creating an animation.** Only invoke `h2v` when the user explicitly asks to export, render, save, or convert the animation to video/MP4. Phrases that should trigger an export run: "export to video", "render as mp4", "convert to video", "save as mp4", "make this a video", "record this animation". A request to "create an animation" by itself is **not** a request to export.

### Before running: check it's installed

`h2v` is not on npm yet; users install from source. Before the first export attempt in a session, check whether the CLI is on `$PATH`:

```bash
command -v h2v
```

If it's missing, tell the user `h2v` is required for video export and offer to install it. The install steps (Node 18+ and `ffmpeg` are prerequisites):

```bash
git clone https://github.com/drewharvey/html-to-video.git
cd html-to-video
npm install
npm install -g .
```

Don't install without confirmation — `npm install -g` mutates the user's global environment.

### h2v-ready by default

Every animation produced by this skill is recordable with no flags because the file template declares both:

- `<meta name="h2v-duration" content="Xs">` — total runtime including the final hold (e.g. last `setTimeout` target + 1–2s end hold = `8s`). This is `h2v`'s highest-priority duration source. If wrong, the recording will cut off mid-animation or include dead time at the end.
- `<meta name="h2v-themes" content="dark,light">` — declares the themes the page supports. The first listed is the default (no attribute on `<html>`); for other passes `h2v` sets `data-theme="<name>"` on `<html>`, which matches the palette selectors. Match this list to whatever themes the animation actually defines.

The controls bar in the template has `data-h2v-hide`, so `h2v` automatically hides it during capture without any file edits.

### Running the export

Only when the user asks:

```bash
h2v export path/to/animation.html
```

Common variants:

- `h2v export --theme all file.html` — one MP4 per declared theme (e.g. `file-dark.mp4`, `file-light.mp4`)
- `h2v export --theme light file.html` — single named theme (must be declared in the meta)
- `h2v export --width 1920 --height 1080 file.html` — non-default viewport (default 1280×720)
- `h2v export --duration 12s file.html` — override the meta tag (rarely needed)

Default output is `./output/<basename>.mp4`. After the run, tell the user where the file landed.

Wall-clock recording time is roughly `animation duration × slowdown` (default 10×), so an 8s animation takes ~80s to record. Don't poll or interrupt it.

### Recording-only styling

If something should be hidden or styled differently *only* during capture (e.g. a debug overlay, an FPS meter, a watermark you want for normal viewing but not the video), use `data-h2v-hide` on the element, or condition on the `data-h2v-recording` attribute that `h2v` sets on `<html>` during capture:

```css
html[data-h2v-recording] .debug-overlay { display: none; }
```

## Previewing the output

After producing an artifact — a new HTML animation file or an exported MP4 — open it for the user, with care so iteration doesn't spam duplicate browser tabs.

### When to auto-open

- **Single newly created file (HTML or MP4):** open it.
- **Edit to an existing file:** do not re-open. The user almost certainly already has it open; just remind them to refresh.
- **Multiple files in one run** (a bundle export, `--theme all` producing 2+ MP4s, a batch of HTML files): do not open any automatically. When the run finishes, ask the user with three options: *open all / open the first / none*. Yes/no is wrong here — six themes shouldn't mean six windows on a "yes."

### How to open

Use the platform-appropriate command:

- macOS: `open <path>`
- Linux: `xdg-open <path>`
- Windows: `start <path>`

If the command fails (headless session, no default app registered, SSH without a display), print the error message **and** the absolute path of the file so the user can open it manually. Do not swallow the failure silently — the user needs to know both that the auto-open didn't work and where the file actually is.

### When the user opted out

Skip auto-open entirely if the request includes any of: "don't open", "just save", "no preview", "skip preview". In those cases, print the file path(s) when done so the user can open them on their own terms.

## Common mistakes to avoid

- **Too much text.** If you're writing a sentence, stop. Use a short label or nothing.
- **Everything appearing at once.** Stagger everything. Even 60ms offsets between items transforms "dumped on screen" into "choreographed."
- **No end state.** The animation must land in a visually balanced final state and hold. Don't let it feel like it stopped mid-thought.
- **Flat layouts.** Use surface layers, borders, and shadows for depth. A card on a background beats content floating in space.
- **Inconsistent border-radius.** Pick a scale and use it everywhere.
- **No visual hierarchy.** Something must be the focal point. Size, color, and position establish what the viewer sees first.
- **Hardcoded colors.** Every hex value is a CSS variable. No exceptions.
- **Missing hover states.** Buttons need `:hover` feedback.
- **Forgetting the theme toggle.** Every animation supports light and dark mode.

## Process

1. Read the request and identify the story arc
2. Choose a palette (use the default or adapt to project context)
3. Plan the phases: what appears when, what changes, what resolves
4. Write HTML structure → CSS styles → JS timing
5. Walk through the animation mentally, second by second: does every moment have purpose? Is there dead time? Does anything compete for attention?
6. Include controls bar and theme toggle
7. Default to dark theme
8. Set the `h2v-duration` meta tag to the total runtime (last animation event + end hold)
9. Open the file for the user (single new file → auto-open; edits → tell them to refresh; multiple → ask)
